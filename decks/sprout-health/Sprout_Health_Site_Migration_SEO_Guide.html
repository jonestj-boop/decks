Site Migration SEO Best Practices

A practical guide for protecting organic traffic and rankings through a website or platform migration.

Prepared for: Sprout Health team Use: Pre-launch, launch, and post-launch reference

A migration is one of the few moments where a site can lose a large share of its organic traffic overnight, and one of the few that is almost entirely preventable. The traffic, rankings, and authority you have already earned live at specific URLs. The entire goal of an SEO-safe migration is to carry that equity from the old URLs to the new ones without losing the signals Google and AI search engines rely on. This guide walks your team through how to do that, before, during, and after launch.

⚠️
The single biggest risk: changing URLs without redirecting them.

If a URL changes and the old one is not redirected to its new equivalent, Google treats the old page as gone (404) and the ranking equity is lost. Most "we migrated and traffic tanked" stories come down to missing, wrong, or lazy redirects.

The golden rules

Keep URLs identical when you can

The safest migration changes the platform but not the URLs. Only change slugs when there is a real reason.

Every old URL gets a 301

Permanent (301) redirect from each old URL to its closest new equivalent. One to one, relevant, not all to the homepage.

Preserve on-page elements

Titles, meta descriptions, H1s, headings, image alt text, canonicals, and structured data should carry over.

Benchmark before you touch anything

Snapshot current traffic, rankings, and indexed pages so you can measure impact and catch problems fast.

Never let staging get indexed

Block the staging/dev site from search engines, then make sure that block is removed at launch.

Launch low-traffic, then monitor

Go live during a slower period and watch Search Console closely for the first few weeks.

Phase 1 — Before you migrate

Inventory

1. Crawl and list every current URL

Use a crawler (Screaming Frog, Sitebulb) plus your XML sitemap and a Search Console export to build a complete list of every live URL: pages, blog posts, landing pages, images, PDFs. You cannot redirect what you have not inventoried.

Benchmark

2. Snapshot current performance

  • Export top pages and queries from Google Search Console (last 12 months).
  • Export organic traffic and top landing pages from GA4.
  • Record current keyword rankings for your priority terms.
  • Save current Core Web Vitals / page speed scores.

This is your "before" picture. Without it you cannot tell whether the migration helped, hurt, or held steady.

Map

3. Build the URL redirect map (the core deliverable)

A simple spreadsheet: every old URL in one column, its new URL in the next. Rules:

  • Map to the most relevant equivalent page, not the homepage.
  • Prioritize your highest-traffic and highest-ranking pages first (from step 2).
  • If a page is being removed, redirect it to the closest related page.
  • Flag any URL whose slug is changing so it gets extra QA.
Preserve

4. Carry over on-page SEO and content

  • Content parity: migrate the full content, do not silently trim or summarize pages.
  • Metadata: title tags, meta descriptions, H1/H2/H3 structure.
  • Images: keep alt text and, ideally, filenames.
  • Structured data / schema (Article, FAQ, Organization, etc.).
  • Canonical tags pointing to the correct new URLs.
  • Internal links: update them to point to the new URLs directly, not through redirects.
Stage

5. Build on staging, blocked from search

Develop the new site on a staging environment that is noindexed (via robots meta tag or HTTP auth). Do not use a robots.txt disallow as your only protection, and never copy that disallow to production by accident at launch.

Phase 2 — Launch day

Cutover checklist
  • Implement all 301 redirects from the map (permanent, not 302 temporary).
  • Confirm redirects are single hops, no chains (A→B→C) or loops.
  • Remove the staging noindex / crawl block from production. (This is the most commonly forgotten step.)
  • Confirm robots.txt on the live site allows crawling and points to the sitemap.
  • Generate and upload an updated XML sitemap with the new URLs only.
  • Verify HTTPS everywhere and that http redirects to https.
  • Confirm one consistent format: www vs non-www and trailing-slash convention.
  • Confirm canonical tags resolve to live, 200-status new URLs.
  • Check that analytics and tracking (GA4, GTM, conversion pixels) are installed on the new site.

Phase 3 — Right after launch

First 24–72 hours
  • Verify the site in Google Search Console (and Bing Webmaster Tools). If the domain or property type changed, set up the new property.
  • Submit the new XML sitemap in Search Console.
  • Use URL Inspection on a few key pages and request indexing.
  • Re-crawl the live site (Screaming Frog) to catch 404s, broken internal links, redirect chains, and any pages accidentally left noindexed.
  • Spot-check 10–20 of your top old URLs in a browser and confirm each 301s to the right new page.
  • Keep the old XML sitemap available temporarily so Google recrawls the old URLs and discovers the redirects faster.
Weeks 1–8
  • Monitor Search Console Coverage / Pages report for spikes in errors or excluded pages.
  • Watch the 404 report and add redirects for anything missed.
  • Track rankings and organic traffic against your benchmark.
  • Expect a temporary dip while Google reprocesses the site. A well-executed migration usually recovers within a few weeks. A sustained drop signals a redirect or indexing problem to investigate.
✅
Expect a short, shallow dip, not a cliff.

Some fluctuation in the first few weeks is normal as search engines recrawl and reindex. The goal is a small dip that recovers, not a large drop that lingers. If traffic falls sharply and stays down, treat it as a redirect/indexing bug and audit immediately.

💡
If the blog stays on WordPress:

If only part of the site moves and the blog remains on WordPress, the lowest-risk path is to keep blog URLs unchanged. New content can be loaded into WordPress as drafts ahead of time and published when ready, no redirects required for those posts.

Common mistakes to avoid

Do

  • Use 301 (permanent) redirects, one hop, to the most relevant page
  • Keep slugs identical unless there is a real reason to change
  • Benchmark before, monitor after
  • Update internal links to the new URLs directly
  • Resubmit sitemaps and re-verify in Search Console

Don't

  • Redirect everything to the homepage
  • Use 302 (temporary) redirects for a permanent move
  • Launch with the staging noindex still on
  • Leave redirect chains or loops in place
  • Change URLs and content at the same time without a map
  • Forget to migrate metadata, alt text, and schema
🚩
The one to triple-check:

Confirm the redirect map covers 100% of the old URLs, that each is a 301, and that the staging crawl block is removed at launch. These three account for the large majority of migration traffic losses.

Quick pre-launch checklist

  • Full crawl + URL inventory complete
  • Performance benchmark captured (GSC, GA4, rankings, speed)
  • Old → new redirect map covers every URL, prioritizing top pages
  • On-page elements preserved (titles, meta, H1s, alt, schema, canonicals)
  • Internal links updated to new URLs
  • Staging noindexed during build
  • 301s built and tested on staging
  • Updated XML sitemap ready
  • HTTPS, www/non-www, trailing-slash conventions set
  • Analytics and tracking installed on new site
  • Launch scheduled for a lower-traffic window
  • Post-launch monitoring plan and owner assigned
