🚨 P1 — Critical · Fix First
P1-A
Homepage Mobile Speed
Score 38/100 · LCP 9.6s · Target: LCP <2.5s · Score 70+
⏰
Why this is P1: Google uses mobile-first indexing. A 9.6s LCP means users bounce before the page renders and rankings suffer directly. No content investment compensates for a broken core web vital.
Step-by-Step Tasks
# Task How Done
1 Run baseline audit Go to pagespeed.web.dev, enter homepage URL, run Mobile report. Screenshot and save the result.
2 Identify LCP element PSI will name the exact element causing delay (usually hero image or large text block). Note it before touching anything.
3 Compress and convert hero image to WebP Use Squoosh.app. Target under 100KB. WebP is 30–40% smaller than JPG at the same quality.
4 Prioritize LCP image load Add loading="eager" and fetchpriority="high" to the LCP <img> tag.
5 Lazy load below-fold images Add loading="lazy" to every <img> tag that is not visible on initial load.
6 Defer non-critical JavaScript Any JS not needed for initial render should use defer or async on the script tag.
7 Audit third-party scripts Tag manager, chat widgets, analytics — each adds load time. Defer what you can. Remove what you don't actively use.
8 Re-run PSI after each change Don't batch all changes then measure. Test after each fix so you know exactly what moved the needle.

P1-B
Page Weight & Request Count
8.5 MB · 528 Requests · Target: <1.5 MB · <100 Requests
⏰
Why this is P1: 8.5 MB is roughly 8x what a fast page should weigh. Every extra request adds a server round-trip. This compounds the LCP problem — fixing speed without fixing weight doesn't fully solve it.
Step-by-Step Tasks
# Task How Done
1 Audit all page assets Open Chrome DevTools → Network tab → reload page. Sort by size. List the top 10 largest assets before touching anything.
2 Convert all images to WebP Every PNG or JPG on the site should become WebP. Batch process with Squoosh or ImageMagick.
3 Resize images to display size No image shown at 800px wide needs a 3,000px source file. Resize to the actual display dimensions before uploading.
4 Enable server-side compression Confirm Gzip or Brotli is active on the host. Check with giftofspeed.com/gzip-test.
5 Enable browser caching Static assets (images, CSS, JS) should have long cache-control headers — 1 year for anything with a hash in the filename.
6 Audit and remove unused plugins If on WordPress, deactivate plugins one at a time and recheck the request count. Remove anything not actively used.
7 Set up a CDN If not already on Cloudflare or similar, set it up. CDN serves assets from edge nodes close to the user, cutting TTFB significantly.
8 Trim web fonts Every font weight loaded is another request. Limit to 2 weights per family. Add font-display: swap to prevent invisible text during load.
9 Minify CSS and JS All CSS and JS files should be minified in production. Most build tools handle this automatically. Verify it's actually happening in the Network tab.

P1-C
Schema Markup — 84 Pages With Zero Schema
Target: Schema on 100% of pages · Minimum 3 schema types deployed
⏰
Why this is P1: Schema is how AI engines (ChatGPT, Gemini, Google AI Overviews) identify and cite your content. The roadmap shows 31 brand mentions today. Schema is the fastest lever to increase that number.
Schema Type by Page Type
Homepage
Organization
Name, URL, logo, sameAs social profiles, contactPoint
Product Pages
Product
Name, description, image, offers (price + availability), brand
Blog Posts
Article
Headline, author, datePublished, dateModified, image, publisher
FAQ Sections
FAQPage
Question + Answer pairs. Also enables FAQ rich results in Google SERPs.
All Pages
BreadcrumbList
Helps Google and AI engines understand site structure. Set once via template.
Implementation Tasks
# Task How Done
1 Map page types to schema types Categorize all 84 pages: Homepage, Product, Article, FAQ, About, Contact. Assign the correct schema type to each group.
2 Add Organization schema to homepage JSON-LD format in <head>. Include name, URL, logo, sameAs (Instagram, Facebook, YouTube), contactPoint.
3 Add Product schema to all product pages Use a template so it auto-populates from existing product data. Don't hardcode each page individually.
4 Add Article schema to all blog posts Should also auto-populate via template. Critical for new content getting into AI citations quickly after publish.
5 Add FAQPage schema where applicable Wrap every Q&A pair in Question/Answer markup. This also enables FAQ rich results in Google — free SERP real estate.
6 Add BreadcrumbList schema sitewide Set up once in a page template so it generates automatically on every page. Helps AI engines map site structure.
7 Validate every schema type before deploying Google Rich Results Test + Schema.org Validator. Fix all errors before pushing live.
8 Resubmit sitemap in Google Search Console GSC → Sitemaps → resubmit. Then manually request indexing on the 10 highest-priority pages via URL Inspection.
9 Re-run AI visibility audit at 30 days Check ChatGPT, Gemini, and Google AI Overviews for brand mentions. Current baseline is 31 mentions across 15 pages. Track movement.
Implementation Order
Homepage → Product Pages → Blog Posts → FAQ Pages → All Others
⚠️ P2 — High · Fix Within 30 Days
P2-A
21 Duplicate Title Tags
Target: Every page has a unique, keyword-targeted title tag
⚡
Why this matters: Duplicate titles cause pages to compete against each other for the same keyword (cannibalization). Google picks one to rank and often ignores the others, splitting traffic that should be consolidated.
Step-by-Step Tasks
# Task How Done
1 Export full title tag list In Screaming Frog or Ahrefs Site Audit, export all page titles. Filter for duplicates. This gives you the exact 21 pages.
2 Group duplicates by root cause Are they from a template default ("Infinity Vans")? Product variants? Pagination? Find the pattern before writing fixes.
3 Write unique titles for each page Each title: primary keyword included, 50–60 characters, completely unique. Use the format below as a guide.
4 Fix the template if it's the source If duplicates come from a CMS template pulling the wrong field, fix the template. Otherwise new pages will duplicate again automatically.
5 Implement and verify Push new titles, then re-crawl with Screaming Frog. Confirm zero duplicates remain before moving on.
Title Tag Format Guide
[Primary Keyword] — Infinity Vans
[Primary Keyword] | [Short Descriptor]
Keep all titles between 50–60 characters. Google truncates anything longer.
P2-B
22 Thin-Content Pages
Target: Every indexed page has substantive content — or is removed from index
⚡
Why this matters: Pages under ~300 words with low informational value signal to Google that the site has quality issues. They dilute overall domain authority and can suppress the rankings of your good pages.
Decision Framework — For Each of the 22 Pages
✏️ Expand
The topic has real value but the page is underdeveloped. Add 500+ words of genuinely useful content. Include FAQs, specs, comparisons. Don't pad.
🔀 Consolidate
Two thin pages cover the same topic. 301 redirect the weaker one to the stronger one. Update all internal links pointing to the old URL.
🚫 Noindex
Utility pages (tag archives, author pages, search results) that can't be meaningfully expanded. Add noindex meta tag.
Step-by-Step Tasks
# Task How Done
1 Pull the list of thin pages Screaming Frog or Ahrefs: filter for indexed pages under 300 words. Cross-reference with GSC to see if they get any impressions.
2 Categorize each page: Expand, Consolidate, or Noindex Use the decision framework above. Document your decision for each of the 22 pages in a spreadsheet before touching anything.
3 Expand pages worth keeping Add 500+ words of useful content. FAQs, specs, how-it-works sections. The content should earn its place — no padding.
4 Consolidate redundant pages Set up 301 redirects from the weaker URL to the stronger one. Update all internal links that pointed to the old URL.
5 Noindex utility pages Add <meta name="robots" content="noindex"> to tag archives, author pages, search result pages.
6 Verify in GSC after 2 weeks Check that noindexed pages are dropping from the index. Check that expanded pages are gaining impressions. GSC → Coverage report.
📋 P3 — Ongoing · Fix Within 60–90 Days
P3-A
29 Pages With Weak Internal Linking
Target: Every page has at least 3 relevant internal links pointing to it
💡
Why this matters: Internal links pass authority between pages and tell Google which pages matter. Orphan and weakly-linked pages don't rank well regardless of content quality. This becomes more urgent as 120 new articles go live — every new page needs to connect to the existing site.
Step-by-Step Tasks
# Task How Done
1 Identify the 29 weak pages Ahrefs → Internal pages sorted by number of inbound inlinks. Flag anything with fewer than 3 internal links pointing to it.
2 Map linking opportunities For each weak page, find 3–5 existing pages that could naturally link to it. Note the anchor text you'd use for each.
3 Add contextual links in body copy Edit source pages to include a natural link. Use descriptive anchor text (e.g., "sprinter van conversion kits"), never "click here."
4 Add to navigation or hub pages If a page is important enough to exist, it should be reachable from a category or hub page. Update nav or hub pages accordingly.
5 Build a hub page if none exists If a cluster of pages has no logical parent, create one (e.g., "Sprinter Van Conversion Kits — Complete Guide") that links to all related pages.
6 Build this into the content publishing checklist Every new article should link to at least 3 existing pages before it goes live. This prevents the problem from growing as 120 new articles publish.
Publishing Checklist Addition
✓ New article links to at least 3 existing pages using descriptive anchor text
✓ At least 2 existing pages have been updated to link to the new article
✓ New article is reachable from its category or hub page
P3-B
16 Broken Internal Links
Target: Zero broken internal links · Ongoing monitoring in place
💡
Why this matters: Broken links waste crawl budget, create dead ends for users, and generate 404 errors that accumulate over time. With 120 new articles on the way, new broken links are likely unless monitoring is set up now.
Step-by-Step Tasks
# Task How Done
1 Pull full broken link list Screaming Frog: Crawl → Response Codes → 4XX. Export source URLs and the broken destination URLs together.
2 Check if the destination page moved If the page exists under a new URL, update the link to point there. If a redirect already exists, verify it resolves cleanly (no redirect chains).
3 Redirect or repoint if page is gone Create a 301 redirect to the most relevant existing page, or update the link in the source page to point to something appropriate.
4 Remove links with no good destination If there's no relevant page to redirect to, remove the link from the source page copy entirely. A missing link is better than a broken one.
5 Set up ongoing monthly monitoring Add a Screaming Frog or Ahrefs broken link alert to run monthly. Broken links will accumulate again without monitoring — especially as new content publishes.
