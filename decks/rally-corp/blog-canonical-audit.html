The headline finding

A revision of what we showed you in the growth roadmap, after auditing the full catalog.

The March fix is implemented client-side — and that's why it's only working 3% of the time. Every post on blog.rallycorp.com still ships <link rel="canonical" href="https://blog.rallycorp.com/<slug>/"> in the initial HTML response — a self-canonical that tells Google "I am the original." A JavaScript snippet labelled "Canonical Manager (Ghost-safe via og:url)" then runs on DOMContentLoaded, removes that tag, and inserts a corrected one pointing at www.rallycorp.com/blog/<slug>/. Google rendered the JS for 3 posts and the fix took effect. For the other 72 posts, Google decided canonical before rendering — or the indexed snapshot predates the fix — and the subdomain copy is what's in the index. Google's John Mueller has been explicit for years: canonical tags must be in the initial server response, not injected by JavaScript.

Why this happens — the JS-canonical antipattern

The Ghost-hosted blog ships every post with this in the <head>:

<link rel="canonical" href="https://blog.rallycorp.com/<slug>/">
<meta property="og:url" content="https://blog.rallycorp.com/<slug>/">
<!-- Canonical Manager (Ghost-safe via og:url) -->
<script>
  // Removes the canonical above, then injects:
  // <link rel="canonical" href="https://www.rallycorp.com/blog/<slug>/">
  document.addEventListener("DOMContentLoaded", setCanonicalFromOgUrl);
</script>

Google's indexing pipeline has two stages: crawl (fetches the HTML) and render (executes JavaScript). The render stage is queued separately and can lag by hours, days, or never run at all if Google deprioritises a page. Whatever canonical Google sees during the first stage often wins, especially when the page is one of many similar URLs. Result: a JS-rewritten canonical is a coin flip, and the coin came up tails on 32% of the catalog.

The fix is to render the correct canonical in the initial HTML response. In Ghost, that's a one-line theme edit — change default.hbs so the canonical_url helper outputs https://www.rallycorp.com/blog/{{slug}}/ instead of the post's own URL.

Recommended actions

In order of leverage. Action #1 stops the bleeding; #2 and #3 reclaim the equity already in Google's index.

1
Move the canonical from JavaScript to the Ghost theme's server-rendered <head>.
Edit the Ghost theme so <link rel="canonical"> is emitted in the initial HTML response with the rallycorp.com/blog/<slug>/ value — not the page's own URL. Remove the "Canonical Manager" script entirely. This is a one-line theme change and is the only intervention that fixes the root cause. Without it, the other actions will keep regressing as Google re-crawls.
2
301-redirect blog.rallycorp.com/<slug>/ to rallycorp.com/blog/<slug> at the edge.
After the canonical is fixed in HTML, layer a 301 on top. A redirect is a stronger signal than a canonical tag — it forces Google to recrawl, transfer equity from the 30 duplicate-indexed posts, and drop the subdomain copies from the index. The 3 "JS render caught" posts already partially benefit from this pattern; rolling it out across all 94 slugs reclaims the rest.
3
Submit a rallycorp.com sitemap that enumerates every /blog/<slug> URL.
For the 42 posts where Google has never crawled the main blog URL, a sitemap submission in Search Console (under the rallycorp.com property) makes those URLs discoverable. Combined with action #2, this is how the equity sitting on the subdomain actually lands at the destination.
4
Internal-link audit: replace every blog.rallycorp.com link with the rallycorp.com equivalent.
If your own navigation, footers, or category pages link to the subdomain, you're voting against your own canonical on every crawl. We can produce the full link-replacement map as a deliverable.
5
Triage the 19 not-indexed posts on content quality.
"Crawled — currently not indexed" usually means thin content, low uniqueness, or no demand signal. Once the canonical issue is resolved, these are the next-highest-leverage cleanup — either consolidate them into stronger pillar content or rewrite for a specific keyword target.
